# Qwen3-TTS로 캐릭터 플랫폼에 목소리 시스템 붙이기: VoiceDesign, CustomVoice, Base, 실시간 추론까지

캐릭터 채팅 플랫폼에 음성을 붙이려면 단순히 "텍스트를 읽어주는 TTS" 하나만 있으면 되는 것이 아니다.
캐릭터별 목소리를 어떻게 만들지, 그 목소리를 어떻게 계속 유지할지, 응답이 길어질 때 얼마나 빨리 첫 음성이 나와야 하는지, 스트리밍 재생은 어떻게 할지, 음성 샘플을 받아서 복제할지 같은 문제가 한꺼번에 따라온다.

이 글은 **QwenLM의 공식 `Qwen3-TTS` 저장소와 공식 문서**를 기준으로, 캐릭터 플랫폼에 `qwen-tts`를 도입할 때 필요한 내용을 최대한 자세하게 정리한 기록이다.

이 글에서 다루는 범위:

- `Qwen3-TTS`가 제공하는 모델 계열과 역할
- `VoiceDesign`, `CustomVoice`, `Base`를 어떤 상황에 써야 하는지
- 캐릭터 플랫폼에 필요한 목소리 생성, 합성, 재사용, 실시간 추론 설계
- 로컬 패키지 기반 자가호스팅과 `DashScope` 실시간 API의 차이
- 2026년 4월 6일 기준 공식 문서가 말하는 제약 사항

---

## 1. Qwen3-TTS는 어떤 모델인가

`Qwen3-TTS`는 Qwen 팀이 2026년 1월 22일 공개한 음성 생성 계열이다.
공식 README 기준으로 다음 특징을 강조한다.

- 10개 주요 언어 지원
- 스트리밍과 비스트리밍 음성 생성 지원
- 자연어 지시 기반 음색, 감정, 억양 제어
- 짧은 샘플을 바탕으로 한 voice clone
- 전용 tokenizer를 통한 음성 encode/decode

특히 공식 README는 `Dual-Track hybrid streaming generation architecture`를 언급하면서,
**한 글자만 입력돼도 첫 오디오 패킷을 낼 수 있고, 종단 지연이 최저 97ms까지 가능하다**고 설명한다.
이건 "실시간 캐릭터 대화" 같은 시나리오에 Qwen3-TTS가 분명히 겨냥되어 있다는 뜻이다.

하지만 여기서 중요한 점이 하나 있다.

**모델 자체가 실시간 친화적이라는 것**과 **개발자가 서비스에서 그 성능을 안정적으로 재현하는 것**은 다른 문제다.

공식 문서 기준으로 보면, `Qwen3-TTS`는 크게 두 가지 경로로 쓸 수 있다.

1. `qwen-tts` Python 패키지로 직접 로컬/서버에서 돌리기
2. `DashScope`의 Qwen3-TTS 실시간 API 사용

그리고 2026년 4월 6일 현재 공식 README는 `vLLM-Omni`에 대해
**"지금은 offline inference만 지원하며 online serving은 later"** 라고 적고 있다.
즉, 현재 시점에 "완성형 온라인 서빙을 vLLM만으로 바로 처리한다"는 기대는 아직 이르다.

## 2. 공식 모델 계열 정리

공식 README가 공개한 주요 모델 계열은 아래와 같다.

| 모델 | 핵심 역할 | 스트리밍 | 지시문 제어 |
|------|------|------|------|
| `Qwen3-TTS-12Hz-1.7B-VoiceDesign` | 설명문으로 새 목소리 설계 | 지원 | 지원 |
| `Qwen3-TTS-12Hz-1.7B-CustomVoice` | 미리 준비된 화자 집합에서 안정적 출력 | 지원 | 지원 |
| `Qwen3-TTS-12Hz-1.7B-Base` | 참조 음성 기반 voice clone | 지원 | 문서상 핵심은 clone |
| `Qwen3-TTS-12Hz-0.6B-CustomVoice` | 더 가벼운 CustomVoice | 지원 | 문서상 별도 표기 없음 |
| `Qwen3-TTS-12Hz-0.6B-Base` | 더 가벼운 Base | 지원 | 문서상 별도 표기 없음 |
| `Qwen3-TTS-Tokenizer-12Hz` | 음성 encode/decode용 tokenizer | 해당 없음 | 해당 없음 |

캐릭터 플랫폼에서 중요한 것은 결국 세 가지다.

- **목소리를 새로 설계할 것인가**
- **정해진 안정적인 화자 세트로 운영할 것인가**
- **샘플 음성으로 특정 목소리를 복제할 것인가**

이 세 질문에 각각 `VoiceDesign`, `CustomVoice`, `Base`가 대응한다.

## 3. 세 모델을 플랫폼 관점에서 이해하기

### 3-1. `CustomVoice`: 가장 운영 친화적인 기본형

공식 README 기준으로 `CustomVoice`는 이미 준비된 화자 집합을 제공한다.
예를 들어 `Vivian`, `Ryan`, `Aiden`, `Ono_Anna`, `Sohee` 같은 화자가 있고,
각 화자는 대략적인 톤과 원어가 정리돼 있다.

이 모델은 보통 다음 상황에 가장 잘 맞는다.

- 서비스 오픈이 급하다
- 캐릭터 수가 많지 않다
- 목소리 품질보다 운영 안정성이 더 중요하다
- 모든 캐릭터를 완전 독자 음색으로 만들 필요는 없다
- 일단 "목소리 붙은 캐릭터 플랫폼"을 빠르게 만들고 싶다

장점:

- 화자 선택이 단순하다
- 출력 품질이 비교적 안정적이다
- 캐릭터별 기본 음색 템플릿을 빨리 만들 수 있다
- `speaker + instruct` 조합으로 감정, 말투, 상황 연기를 덧입히기 쉽다

한계:

- 근본적으로는 정해진 화자 집합 안에서 움직인다
- 캐릭터가 많아질수록 음색 중복 체감이 생길 수 있다
- 세계관이 강한 플랫폼에서는 "이 캐릭터만의 고유성"이 부족할 수 있다

즉, `CustomVoice`는 캐릭터 음성 시스템의 **MVP용 기본 선택지**에 가깝다.

#### 이런 식으로 쓰면 좋다

- 메인 캐릭터 8명 이하
- 각 캐릭터에 화자 1명 고정
- 감정 프리셋만 별도 운영
- 예:
  - 기본 상태: `calm, friendly, medium speed`
  - 전투 상태: `sharp, urgent, fast`
  - 친밀 상태: `soft, intimate, lower volume feel`

실제로는 음량을 직접 지정하는 것이 아니라, 그런 느낌이 나게 하는 instruction을 플랫폼 레벨에서 템플릿화하는 방식이 더 현실적이다.

---

### 3-2. `VoiceDesign`: 캐릭터 제작 툴에 가깝다

`VoiceDesign`은 공식 README 기준으로 **자연어 설명문으로 목표 목소리를 설계**하는 모델이다.

이 모델은 "캐릭터의 목소리를 플랫폼 운영자가 새로 만든다"는 개념에 가깝다.
즉, 단순히 읽는 엔진이라기보다 **캐릭터 음색 디자이너** 역할을 한다.

이 모델이 빛나는 상황:

- 캐릭터 생성형 플랫폼
- 유저가 직접 캐릭터를 만들고 음성 설정을 넣는 서비스
- 캐릭터 설정문이 이미 풍부한 서비스
- 게임/비주얼노벨형 세계관에서 역할별 음색 차별화가 중요한 경우

예를 들어 캐릭터 카드에 아래 같은 정보가 있다면:

- 성별
- 대략 연령대
- 언어
- 말 빠르기
- 긴장/차분/냉소/장난기 같은 정서 속성
- 발성 특징
- 지역 억양 힌트

이걸 `VoiceDesign`용 지시문으로 변환할 수 있다.

예시 개념:

```text
여성, 20대 초반, 차갑고 또렷한 고음역.
문장 끝은 짧게 끊고, 감정을 잘 드러내지 않지만
친한 상대 앞에서는 미세하게 부드러워진다.
속도는 약간 빠르고 발음은 매우 선명하다.
```

장점:

- 캐릭터 정체성에 맞춘 음성 설계가 가능하다
- 플랫폼의 캐릭터 메타데이터와 잘 결합된다
- "캐릭터마다 다른 목소리"라는 인상을 강하게 줄 수 있다

한계:

- 같은 설명문이라도 완전히 고정된 화자처럼 느껴지지 않을 수 있다
- 장기 운영에서는 "캐릭터 음색이 약간씩 흔들리는가"를 직접 검증해야 한다
- 설명문 설계 품질이 낮으면 캐릭터성이 흐려질 수 있다

결론적으로 `VoiceDesign`은 **캐릭터 제작 단계**에서 강하다.
하지만 실서비스에서 한 캐릭터가 수백, 수천 줄을 말해야 한다면 이것만으로 끝내기보다,
다음에 설명할 `Base`와 함께 묶는 전략이 더 좋다.

### 3-3. `Base`: 플랫폼에서 가장 중요한 "고정 캐릭터화" 수단

`Base`는 공식 README 기준으로 **3초 참조 음성 기반의 rapid voice clone**을 지원한다.
새 텍스트를 읽게 하려면 `ref_audio`와 `ref_text`를 제공하면 된다.

이 모델이 중요한 이유는,
캐릭터 플랫폼에서 진짜 어려운 것은 "한 번 예쁜 목소리"가 아니라
**"그 목소리를 같은 사람처럼 계속 유지하는 것"** 이기 때문이다.

`Base`는 바로 그 지점에 들어간다.

공식 문서가 제시하는 가장 실전적인 흐름은 이렇다.

1. `VoiceDesign`으로 캐릭터 기준 음성 샘플을 만든다
2. 그 샘플을 `Base`에 넣어 `voice_clone_prompt`를 만든다
3. 이후 실제 대사 생성은 `generate_voice_clone`으로 계속 처리한다

즉,

- `VoiceDesign` = 목소리 기획/설계
- `Base` = 그 설계를 반복 사용 가능한 화자로 굳히는 단계

이 조합은 캐릭터 플랫폼에 특히 잘 맞는다.

#### 왜 `voice_clone_prompt`가 중요하나

공식 README는 같은 참조 프롬프트를 반복 사용할 경우,
매번 특징을 다시 뽑지 말고 `create_voice_clone_prompt`로 한 번 생성해 재사용하라고 설명한다.

이건 플랫폼 측면에서 매우 중요하다.

- 캐릭터 A의 음성 prompt를 미리 캐싱할 수 있다
- 대화가 이어질 때마다 재계산 비용을 줄일 수 있다
- 음성 일관성도 더 관리하기 쉬워진다

실무적으로는 캐릭터 DB에 아래 개념이 들어간다.

```text
character_id
voice_mode = base_clone
voice_model = Qwen3-TTS-12Hz-1.7B-Base
voice_reference_audio_path
voice_reference_text
voice_clone_prompt_cache_key
default_language
default_tts_style_preset
```

## 4. 캐릭터 플랫폼에서 어떤 모델을 어떻게 고를까

이 절부터는 공식 README가 공개한 기능을 바탕으로, **캐릭터 플랫폼 운영 관점에서 내가 정리한 설계 추론과 추천안**이 포함된다.

### 4-1. 가장 추천되는 운영 전략

캐릭터 플랫폼에서는 아래 3단 구조가 가장 현실적이다.

#### 전략 A: 빠른 출시형

- 기본 운영: `CustomVoice`
- 캐릭터 수: 적음
- 목적: 빠른 MVP, 저위험

이 경우 캐릭터별로 speaker만 고정하고,
상황별 감정은 instruction 템플릿으로 관리한다.

#### 전략 B: 캐릭터성 강화형

- 캐릭터 설계: `VoiceDesign`
- 실시간 발화: `VoiceDesign` 그대로 또는 일부 `Base` 전환

이 경우 창작 자유도는 높지만,
캐릭터가 장기간 같은 목소리로 느껴지는지 반드시 QA가 필요하다.

#### 전략 C: 서비스 운영 최적형

- 캐릭터 설계: `VoiceDesign`
- 고정 화자화: `Base`의 `voice_clone_prompt`
- 실시간 발화: `Base`

이 전략이 가장 "캐릭터 플랫폼다운" 구조다.
처음엔 음색을 만들고, 이후에는 그 음색을 재사용 가능한 캐릭터 보이스로 굳혀서 쓴다.

---

### 4-2. 모델 선택 표

| 상황 | 추천 모델 | 이유 |
|------|------|------|
| 데모를 빨리 만들고 싶다 | `CustomVoice` | 바로 화자 선택 가능 |
| 운영자가 캐릭터 목소리를 설계하고 싶다 | `VoiceDesign` | 설명문 중심 설계 가능 |
| 유저 음성이나 샘플 음성으로 캐릭터화하고 싶다 | `Base` | 참조 음성 기반 clone |
| 장기적으로 동일 캐릭터를 계속 말하게 해야 한다 | `VoiceDesign` + `Base` | 설계와 고정화를 분리 가능 |
| 서버 비용이 빡빡하다 | `0.6B` 계열 검토 | 더 가벼운 계열 |
| 품질을 우선한다 | `1.7B` 계열 우선 | 더 높은 표현력 기대 |

---

## 5. 캐릭터 플랫폼에 필요한 음성 요소를 분해해 보기

캐릭터 플랫폼에서 "목소리 기능"은 사실 여러 서브시스템의 조합이다.

### 5-1. Voice Creation

캐릭터의 목소리를 처음 만드는 단계다.

방법은 세 가지다.

1. `CustomVoice`에서 기본 화자 선택
2. `VoiceDesign`으로 설명문 기반 생성
3. `Base`로 참조 음성 기반 복제

이 단계에서는 아래 메타데이터를 함께 저장해야 한다.

- 사용 모델명
- 언어
- 기본 speaker 또는 지시문
- 참조 음성 파일 경로
- 참조 텍스트
- 샘플 생성 결과
- 운영자 승인 여부

이걸 저장하지 않으면,
나중에 "이 캐릭터 목소리 왜 바뀌었지?"라는 문제가 생겼을 때 원인 추적이 불가능해진다.

---

### 5-2. Voice Synthesis

실제 대사를 음성으로 바꾸는 단계다.

입력:

- 캐릭터 ID
- 출력할 대사 텍스트
- 대사 언어
- 감정/상황 태그
- 현재 세션 상태

출력:

- wav/pcm/audio chunk
- 샘플레이트
- 필요 시 metadata

플랫폼에서는 단순히 `text -> wav`로 끝나지 않는다.
다음 요소도 함께 고려해야 한다.

- 새 응답이 오면 이전 음성을 중단할 수 있어야 함
- 긴 답변은 문장 단위로 잘라 먼저 재생할 수 있어야 함
- 모바일/웹 클라이언트가 스트리밍 재생 가능해야 함
- 동일 캐릭터라도 장면별 감정 가중치를 줄 수 있어야 함

---

### 5-3. Real-Time Inference

실시간 추론은 "모델이 스트리밍 가능하냐"보다 더 넓은 문제다.
플랫폼 레벨에서는 보통 아래 조건을 만족해야 실시간처럼 느껴진다.

- 첫 음성 시작이 빠를 것
- 문장 도중 끊김이 적을 것
- 사용자가 새 메시지를 치면 이전 음성이 즉시 멈출 것
- 네트워크 지연이 있어도 버퍼가 안정적일 것

`Qwen3-TTS` 공식 README는 모델 차원에서는 스트리밍을 지원한다고 설명한다.
다만 개발자가 자가호스팅으로 그 경험을 만들려면,
아래 계층을 모두 직접 설계해야 할 수 있다.

- 텍스트 스트리밍
- 문장 분할기
- TTS 요청 큐
- 오디오 chunk 전송
- 브라우저/앱 재생 버퍼
- 중간 취소와 덮어쓰기

즉, 모델이 전부 해결해주지 않는다.

---

## 6. 실제 서비스 구조는 어떻게 짜는가

캐릭터 플랫폼 기준 권장 구조는 아래와 같다.

```text
[사용자]
   ↓
[캐릭터 채팅 프론트]
   ↓ WebSocket
[대화 오케스트레이터]
   ├── LLM 응답 생성
   ├── 문장 분할
   ├── TTS 작업 큐
   └── 세션 취소 제어
          ↓
[Qwen3-TTS 서비스]
   ├── CustomVoice
   ├── VoiceDesign
   └── Base / voice_clone_prompt cache
          ↓
[오디오 스트림]
   ↓
[웹/앱 플레이어]
```

핵심은 `Qwen3-TTS`를 직접 프론트가 두드리게 하지 말고,
중간에 **대화 오케스트레이터 계층**을 두는 것이다.

이 계층이 해야 하는 일:

- 캐릭터별 음성 정책 선택
- 긴 답변을 문장 단위로 자르기
- 이전 음성 취소
- 감정 태그를 instruction으로 변환
- 음성 캐시 여부 판단
- 장애 발생 시 텍스트만 우선 표시

---

## 7. `VoiceDesign`, `CustomVoice`, `Base`를 서비스 흐름에 배치하기

### 7-1. `CustomVoice` 중심 구조

가장 단순하다.

```text
캐릭터 프로필
  → 기본 speaker 지정
  → 감정별 instruction 템플릿 지정
  → 응답 생성 시 바로 generate_custom_voice 호출
```

좋은 점:

- 구조가 단순하다
- 온보딩이 빠르다
- 운영자가 이해하기 쉽다

아쉬운 점:

- 캐릭터가 많아질수록 목소리 중복감이 커질 수 있다

---

### 7-2. `VoiceDesign` 중심 구조

```text
캐릭터 생성 시
  → 설정문으로 VoiceDesign 샘플 생성
  → 운영자가 샘플 승인
  → 그대로 실시간 대사에도 사용
```

좋은 점:

- 캐릭터 개성이 크다
- 세계관 반영이 쉽다

아쉬운 점:

- 장기 일관성을 직접 검증해야 한다

이 구조는 초기 실험용으로는 좋지만,
상용 운영에서는 `Base`와 함께 쓰는 편이 더 안전하다.

---

### 7-3. `VoiceDesign -> Base` 구조

이게 가장 추천되는 운영 구조다.

```text
캐릭터 생성
  → VoiceDesign으로 기준 음성 샘플 생성
  → Base에서 create_voice_clone_prompt 수행
  → 캐시 저장
  → 실제 발화는 generate_voice_clone로 처리
```

이 구조의 장점:

- 캐릭터 제작 자유도 확보
- 운영 중 음성 일관성 확보
- 반복 호출 시 프롬프트 재계산 감소

즉, **창작과 운영을 분리**할 수 있다.

---

## 8. 실시간 캐릭터챗에 붙일 때 반드시 필요한 기능

### 8-1. 문장 분할기

LLM이 긴 답변을 생성하면,
그 전체를 다 기다렸다가 TTS로 넘기면 첫 음성이 너무 늦어진다.

그래서 보통은:

- 쉼표 기준 임시 분할
- 마침표 기준 확정 분할
- 문장 길이 상한 도입

같은 로직이 필요하다.

이때 중요한 것은 너무 잘게 자르지 않는 것이다.
한두 단어 단위로 잘라 버리면 억양이 부자연스럽다.

---

### 8-2. 취소 가능한 TTS 세션

캐릭터 플랫폼에서는 사용자가 새 메시지를 보내면
이전 응답 음성을 즉시 중단해야 한다.

그래서 TTS 작업마다 아래 정보가 필요하다.

- session_id
- conversation_id
- utterance_id
- cancel_token

이걸 두지 않으면 이전 음성과 새 음성이 겹쳐 재생된다.

---

### 8-3. 감정 매핑 레이어

캐릭터 LLM이 만든 상태값을 TTS instruction으로 바꾸는 레이어가 필요하다.

예:

```json
{
  "emotion": "embarrassed",
  "intensity": 0.7,
  "pace": "fast",
  "relationship_state": "close"
}
```

이를 다음 같은 instruction으로 번역할 수 있다.

```text
조금 당황했지만 귀엽게 숨기려는 톤.
말속도는 약간 빠르게, 문장 끝이 살짝 떨리는 느낌.
친한 상대에게 말하듯 거리감은 가깝게.
```

이 레이어가 있어야 대화 엔진과 TTS 엔진이 느슨하게 결합된다.

---

### 8-4. 캐시 전략

모든 대사를 매번 새로 합성하면 비용과 지연이 커진다.

캐릭터 플랫폼에서는 다음 캐시가 특히 유효하다.

- `voice_clone_prompt` 캐시
- 자주 쓰는 고정 문구 캐시
- 로그인 인사, 알림, UI 응답음 캐시

예:

- "어서 와."
- "지금은 조금 바빠."
- "잠깐만 생각해볼게."

이런 문장은 미리 생성해두면 체감 반응 속도를 크게 올릴 수 있다.

---

## 9. 자가호스팅과 API 중 무엇이 더 맞을까

### 9-1. `qwen-tts` 자가호스팅

공식 README는 로컬/서버 사용을 위해 아래 흐름을 제공한다.

```bash
conda create -n qwen3-tts python=3.12 -y
conda activate qwen3-tts
pip install -U qwen-tts
pip install -U flash-attn --no-build-isolation
```

장점:

- 플랫폼 내부 정책에 맞춰 깊게 커스터마이즈 가능
- 캐시, 취소, 세션 제어를 마음대로 설계 가능
- 사용자 데이터와 참조 음성을 사내/자체 인프라에 둘 수 있음

단점:

- 온라인 스트리밍 서빙 레이어를 직접 짜야 할 수 있음
- 운영 난이도와 GPU 관리 부담이 큼

---

### 9-2. `DashScope` 실시간 API

공식 README는 `DashScope API Usage`에서
`Custom Voice`, `Voice Clone`, `Voice Design` 각각에 대해 **real-time API 문서 링크**를 제공한다.

즉, 실시간 서비스 경험을 더 빨리 만들고 싶다면
Qwen 팀이 제공하는 API 계층을 활용하는 것이 자연스러운 선택이다.

장점:

- 실시간 API 경로가 공식적으로 열려 있다
- 자가호스팅보다 빠르게 서비스 프로토타입을 만들 수 있다

단점:

- 외부 API 의존성이 생긴다
- 비용, 지역, 정책, 데이터 처리 이슈를 검토해야 한다

---

### 9-3. `vLLM-Omni`

2026년 4월 6일 기준 공식 README는 `vLLM-Omni`에 대해
**현재는 offline inference만 지원하고 online serving은 추후 지원 예정**이라고 적고 있다.

따라서 지금 시점에 실시간 캐릭터 플랫폼을 만든다면:

- `vLLM-Omni` 단독으로 "즉시 상용형 온라인 서빙"을 기대하기보다는
- `qwen-tts` 직접 서비스화 또는 `DashScope` API 활용 쪽이 더 현실적이다

이건 기술 성능 문제가 아니라, 공식 문서가 밝힌 지원 범위 문제다.

## 10. 플랫폼 기능별 추천 조합

### 10-1. 고정 캐릭터형 연애/롤플레잉 서비스

추천:

- 메인 설계: `VoiceDesign`
- 고정 운영: `Base`
- 보조 캐릭터: `CustomVoice`

이유:

- 메인 캐릭터는 고유성이 중요하다
- 서브 캐릭터는 운영 효율이 더 중요하다

---

### 10-2. 유저 생성 캐릭터 플랫폼

추천:

- 캐릭터 생성 시 `VoiceDesign`
- 유저가 샘플 업로드 가능하면 `Base`
- 반복 사용 캐릭터는 `voice_clone_prompt` 저장

주의:

- 저작권/초상권/목소리 권리 문제
- 타인 음성 무단 복제 방지 정책

---

### 10-3. 방송형 AI 캐릭터

추천:

- 고정 메인 캐릭터는 `Base`
- 상황별 스타일 변경은 instruction 템플릿
- 실시간성이 중요하면 API 기반 스트리밍 경로 우선 검토

이유:

- 방송형은 음색 일관성이 특히 중요하다
- 한 번 만든 목소리가 방송 내내 흔들리면 몰입이 깨진다

---

## 11. 개발 시 주의할 점

### 11-1. "좋은 목소리"와 "좋은 서비스"는 다르다

샘플 하나가 잘 나오더라도,
실서비스에서는 아래가 더 중요하다.

- 50턴 대화 동안 캐릭터성이 유지되는가
- 긴 문장과 짧은 문장 품질 차이가 큰가
- 감정 instruction이 과하면 오히려 부자연스럽지 않은가
- 중간 취소가 자연스럽게 되는가

---

### 11-2. 레퍼런스 음성 관리를 엄격히 해야 한다

`Base`를 쓰면 참조 음성 품질이 곧 서비스 품질이 된다.

최소한 아래 기준이 필요하다.

- 잡음 적음
- 발화가 또렷함
- 캐릭터 톤이 잘 드러남
- 참조 텍스트가 정확히 저장돼 있음

공식 README도 `ref_text`를 함께 넣는 방식을 기본으로 설명한다.

---

### 11-3. Base 모델의 웹 데모는 HTTPS가 사실상 중요하다

공식 README는 `Base` 모델 데모를 원격 브라우저에서 쓸 경우
마이크 권한 문제 때문에 HTTPS 실행을 권장/사실상 요구한다고 적고 있다.

즉, 캐릭터 플랫폼에서

- 유저 음성 업로드
- 브라우저 마이크 입력
- 실시간 voice clone 체험

같은 기능을 붙일 생각이라면, 프런트와 미들웨어의 HTTPS 설계가 필수에 가깝다.

---

## 12. 추천 구현 순서

캐릭터 플랫폼에 Qwen3-TTS를 붙인다면, 아래 순서가 가장 안전하다.

### 1단계: `CustomVoice`로 음성 파이프라인 완성

- 텍스트 생성
- TTS 요청
- 오디오 재생
- 취소 처리

먼저 이 4단계를 완성한다.

### 2단계: `VoiceDesign`으로 캐릭터별 샘플 제작 툴 추가

- 운영자용 음성 설계 UI
- 샘플 비교
- 승인/반려

### 3단계: `Base`로 캐릭터 보이스 고정화

- 참조 음성 저장
- `voice_clone_prompt` 캐시
- 캐릭터별 기본 보이스 확정

### 4단계: 실시간 최적화

- 문장 분할 스트리밍
- 오디오 chunk 전달
- 세션 중단 처리
- 자주 쓰는 문장 캐시

이 순서를 거꾸로 가면, 음성은 화려한데 제품은 불안정한 상태가 되기 쉽다.

## 13. 한 줄 결론

`Qwen3-TTS`는 캐릭터 플랫폼에 목소리 요소를 붙이기에 매우 강한 선택지다.
하지만 가장 중요한 포인트는 단순히 "TTS 품질이 좋다"가 아니다.

진짜 핵심은 다음과 같다.

- `CustomVoice`는 빠른 MVP와 안정적 운영에 좋다
- `VoiceDesign`은 캐릭터 제작 자유도를 높여준다
- `Base`는 캐릭터 목소리를 장기적으로 고정하는 데 핵심이다
- 실시간 서비스 경험은 모델 하나가 아니라 **오케스트레이션 계층**까지 포함해 설계해야 한다

내 기준 추천 조합은 이것이다.

```text
캐릭터 설계는 VoiceDesign,
장기 운영은 Base,
빠른 프로토타입과 보조 캐릭터는 CustomVoice.
```

이렇게 나누면 창작성과 운영성을 같이 잡기 쉽다.
