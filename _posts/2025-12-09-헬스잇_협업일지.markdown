# 12월 9일 협업일지

날짜: 2024-12-09
이름: 안호성
팀명: 4팀

## 오늘 맡은 역할 및 작업 내용

YOLO 학습을 위한 데이터 파이프라인 구축을 이어서 진행했고,
실제 모델 학습 및 Kaggle 제출까지 end-to-end 플로우를 완성한 날이었다.

- YOLO 데이터 파이프라인 고도화
    - 원시 COCO 데이터의 category_id가 불연속적이라는 문제 발견
    - YOLO는 클래스 ID가 0부터 시작하는 연속 정수여야 하므로, YOLO class ID <-> COCO category ID 매핑 JSON을 추가로 생성
    - 해당 매핑 파일을 CocoParser 클래스 내부 로직에 통합하여 변환 과정이 자동으로 처리되도록 구조 개선
- YOLOv8n 학습용 데이터셋 구축
    - COCO -> YOLO TXT 변환 완료
    - data.yaml 구성(train/val 경로, class 수, class 이름 등)
    - YOLO 학습이 가능한 디렉토리 구조 최종 정리
- YOLOv8n 학습(50 epochs) 수행
    - 첫 실험 결과:
        - mAP50: 0.8730
        - mAP50-95: 0.8460
        - 제한된 데이터셋 크기(232개) 대비 높은 수치 확보
- Kaggle 제출용 결과 생성
    - YOLOv8n 모델로 테스트셋 추론 수행
    - Kaggle 제출 포맷(CSV)으로 자동 변환 스크립트 작성
    - Kaggle 최초 제출 결과: 0.73886
- GitHub 커밋 관리
    - COCO parser, 매핑 JSON, YOLO 데이터 생성 코드 등 전체 변경 사항을 GitHub에 즉시 반영하여 팀 전체에 공유

## 오늘 작업 현황

YOLO 데이터 파이프라인 완성, 모델 학습, 제출까지 모든 단계를 성공적으로 연결했다.
모델 검증 수치와 Kaggle 리더보드 점수까지 확보했기 때문에 오늘 목표는 실질적으로 완료한 것이나 마찬가지였다.

## 오늘 협업 중 제안하거나 피드백한 내용

- 팀원들에게 시각화된 박스 이미지를 직접 공유하여 데이터셋 품질 문제(누락/오류 박스)를 명확히 인지시켰다.
- 검증 결과와 Kaggle 점수를 공유하며 현재 데이터로 가능한 baseline 수준을 확보했다는 점을 설명했다.
- category id 매핑 문제를 상세히 설명하고, 추후 모델 실험을 위해선 이 과정이 필수임을 강조했다.

## 오늘 분석/실험 중 얻은 인사이트나 발견한 문제점

- 원시 COCO category id가 불연속적이기 때문에 단순 변환으로는 YOLO 학습이 불가능함을 확인했다.
    - 데이터셋 특성에 맞춘 매핑 전략이 필수.
- annotation 품질이 모델 학습에 직접적 영향을 미친다는 점을 체험했다.
    - 일부 이미지의 박스 누락/오류는 성능의 최대 상한선을 제한할 가능성이 있다.
- 그럼에도 불구하고 mAP50 0.87, LB 0.73 수준이 나온 것은 모델 자체의 성능이 우수하고, 데이터셋 난이도가 낮은 구간도 존재함을 의미한다.
- 향후 제출 점수를 더 끌어올리려면 다음이 필요하다.
    - annotation 자동 검증 및 보정
    - 라벨 오류 수동 정리
    - 추가 데이터 확보

## 일정 지연이나 협업 중 어려웠던 점

- 매핑 JSON 도입 전까지 category 불연속성 문제로 시간이 소요되었다.
- GitHub 활용 부족 문제는 여전히 지속되어 실제 코드 기반 협업이 나에게 편중된 상태이다.

## 오늘 발표 준비나 커뮤니케이션에서 기여한 부분

- 모델 검증 이미지(박스 시각화 결과)와 Kaggle 결과를 팀원에게 공유하여 데이터 문제와 모델 현황을 명확히 전달했다.
